## 프로젝트 배경

제조 현장에서는 불량이 드물게 발생하기 때문에 지도학습을 위한 불량 데이터 확보가 어렵다.

MVTec AD Hazelnut 데이터셋을 활용해, **정상 이미지만으로 학습한 Isolation Forest가 실제로 불량을 탐지할 수 있는지** 검증했다. 이 과정에서 가장 중요한 질문은 "모델이 좋은가"가 아니라 **"추출한 특징 벡터에 탐지에 필요한 정보가 담겨 있는가"** 였다.

---

## 주요 기능

- HSV 색 공간 변환 후 3D Histogram으로 512차원 특징 벡터 추출

- Threshold Grid Search로 최적 임계값 탐색 (F1 기준)
- 차영상(Difference Image) + Morphology + Contour Detection으로 결함 후보 영역 시각화
- Bounding Box 오버레이로 결함 위치 표시

## 핵심 구현 내용

### 1. HSV Color Histogram 특징 추출 — 그리고 그 한계

**구현:** 이미지를 RGB → HSV로 변환 후, H/S/V 채널 각각의 Histogram을 계산하여 정규화된 512차원 특징 벡터 생성

**HSV를 선택한 이유:** RGB는 조명 변화에 민감하다. HSV는 색조(Hue)를 밝기와 분리하여 더 안정적인 색상 표현을 제공하므로 공장 환경의 조명 편차에 덜 영향받을 것으로 판단했다.

**실제 한계:** Histogram은 픽셀의 색상 분포만 담을 뿐 **위치 정보와 공간 구조를 전혀 포함하지 않는다.** Crack, Cut처럼 형태의 변화로 나타나는 결함은 정상 이미지와 Histogram이 사실상 동일하게 나타난다.

---

### 2. Isolation Forest 학습 및 파라미터 설정

**구현:** 정상 이미지의 특징 벡터만 학습하여 Anomaly Score를 계산. 정상에서 멀수록 낮은 점수 부여

**파라미터 선택 근거:**
- `n_estimators=500`: 300 대비 더 세밀한 결정 경계 — 미세 이상 점수 차이 포착 목적
- `max_samples=0.8`: 개별 트리 간 다양성 확보로 일반화 향상
- `max_features=1.0`: 512차원 전체 사용 — 결함 정보 손실 방지

---

### 3. Threshold Grid Search

**구현:** 다양한 Threshold 값에서 Precision, Recall, F1을 계산하여 최적값 탐색

**결과가 보여준 것:** 최적 Threshold 적용 시 F1=0.26이 실질적 상한선. 이 수치는 Threshold 문제가 아닌 **특징 추출 방식 자체의 한계임을 명확히 보여준다.**

## 주요 트러블슈팅

### 문제 1: 불량 70장 전체 미탐지 (FN=70, Recall=0.00)

**문제:** 최적 Threshold를 적용해도 불량 이미지 70장 전체가 정상으로 분류됨

**원인 분석:** Crack, Cut, Hole은 형태(구조)의 변화이며 색상 Histogram에는 반영되지 않는다. 특징 벡터가 탐지에 필요한 정보를 담고 있지 않다면 어떤 모델도, 어떤 Threshold도 이를 분리할 수 없다.

**해결 시도:** Threshold를 낮추어 더 많은 이미지를 이상으로 분류하도록 조정했으나, F1=0.26이 실질적 상한선임을 확인. **모델이 아닌 특징 추출 방식의 변경이 필요하다는 결론 도출**

---

### 문제 2: Print 결함만 부분적으로 탐지 — 이유 분석

**문제:** 4가지 결함 중 Print만 Anomaly Score가 정상과 다소 차이를 보임

**원인 분석:** Print 결함은 잉크 자국으로 **색상 자체가 변한다.** HSV Histogram이 포착할 수 있는 유일한 변화 유형이다. 반면 Crack, Cut, Hole은 표면의 형태 변화이며 색상 분포는 정상과 거의 동일하다. 이를 통해 "색상 기반 특징은 색상이 변하는 결함만 탐지 가능하다"는 명확한 결론을 얻었다.

---

### 문제 3: 차영상 시각화에서 위치 노이즈 과다 발생

**문제:** 정상 참조 이미지와 테스트 이미지의 차이를 계산했을 때, 결함이 아닌 헤이즐넛 위치 변화가 대부분을 차지함

**원인 분석:** 헤이즐넛이 이미지마다 다른 위치에 놓이기 때문에 단순 픽셀 차이 계산 시 전체 외형이 변화로 잡힘

**해결 방법:** Morphology 연산으로 노이즈를 제거하고, Contour Detection으로 면적이 큰 영역만 추출 후 Bounding Box 표시

**결과:** Print 결함에서는 시각화 성공, Crack/Cut은 위치 변화 노이즈로 인해 정확한 위치 표시 한계 존재

## 배운 점

**기술적 인사이트**
- 특징 추출(Feature Engineering)은 모델 성능의 상한선을 결정한다. 특징 벡터에 없는 정보는 어떤 모델도 학습할 수 없다.
- "어떤 특징을 추출하는가"는 "어떤 모델을 쓰는가"보다 더 근본적인 선택이다.
- 성능 지표가 나쁘게 나왔을 때, 모델이나 하이퍼파라미터보다 특징 벡터부터 의심해야 한다.

**실무적 인사이트**
- F1=0.00이라는 결과 자체보다, "왜 0.00인가"를 분석한 것이 진짜 성과다.
- 실패 사례의 원인을 구조적으로 분석하는 능력이 성공 사례를 재현하는 능력보다 더 중요하다.
