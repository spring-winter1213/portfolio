# 버스바 비지도 이상탐지 파이프라인 — PatchCore 기반 OCC

실제 공장 버스바 검사 라인에 비전 AI를 적용한 산업용 이상탐지 시스템.
30개 카메라-검사 조건에서 정상 이미지만으로 학습하여 불량을 자동 감지하며,
v3→v4→v5 실험을 통해 FN을 36에서 4로 감소시킨 개선 과정을 담은 프로젝트.

> 팀 레포지토리: [KDT12-mnvision/KDT12-mnvison](https://github.com/KDT12-mnvision/KDT12-mnvison)
> 이 문서는 해당 팀 프로젝트에서 **장윤진이 담당한 기여 내용**을 정리한 개인 포트폴리오 문서입니다.

## 프로젝트 배경

실제 공장 버스바 검사 라인에 비전 AI를 도입하는 것이 목표였다.
30개 카메라-검사 조건에 대해 불량 이미지를 확보하려 했으나,
**절반 이상의 조건에서 불량 이미지가 0장이었고, 불량이 있는 조건도 평균 3장에 불과했다.**

지도학습은 데이터 부족으로 불가능했고, 정상 이미지만으로 학습하는 OCC(One-Class Classification) 방식이 유일한 현실적 선택지였다.
논문에서 MVTec 벤치마크 기준 AUROC 99.1%를 달성한 PatchCore를 채택하되,
현장 데이터 특성에 맞게 3번의 실험을 통해 파이프라인을 조정했다.

## 주요 기능

- PatchCore 기반 비지도 이상탐지 (정상 이미지만으로 학습)
- 30개 카메라-검사 조건 각각에 대한 독립적 모델 운용

- 배포 가능한 퍼센타일 기반 임계값(Threshold) 적용
- 이상 위치 시각화를 위한 Heatmap 생성

## 핵심 구현 내용

### 1. v3 — 백본 비교 실험 (1,440개 조합)

**무엇을:** ResNet-18, ResNet-50, WideResNet-50-2 세 가지 백본을 30개 조건 × 다수 하이퍼파라미터 조합(총 1,440개)에서 비교

**왜 이렇게 했는가:** PatchCore 논문은 WideResNet-50-2를 권장하지만, 실제 데이터에서도 동일한 경향이 나타나는지 검증이 필요했다. 가정을 그대로 사용하지 않고 직접 실험으로 확인했다.

**결과:**
- WideResNet-50-2: AUROC_V 0.436 (최고)
- ResNet-18: 0.430
- ResNet-50: 0.430
- SoftPatch 전처리 및 Homomorphic 필터 효과 없음 → v4에서 제외

---

### 2. v4 — 하이퍼파라미터 최적화 (1,493개 조합)

**무엇을:** WideResNet-50-2 고정 후 이미지 해상도(256/384), Coreset 비율, kNN 이웃 수를 조합하여 1,493개 실험 진행

**왜 이렇게 했는가:** 백본을 고정하고 나머지 변수를 체계적으로 탐색하는 것이 실험 설계의 원칙이다. 모든 변수를 동시에 바꾸면 무엇이 성능에 영향을 미쳤는지 알 수 없다.

**결과:**
- 조건별 베스트 AUROC_V 평균: 0.466
- 30개 조건 중 8개에서 AUROC 1.0 달성
- 해상도 384 적용 시 recall_v: 0.441 → 0.453
- **문제 발견: v4의 임계값은 테스트 레이블을 보고 사후에 결정 → 실제 배포 불가능**

---

### 3. v5 — 퍼센타일 임계값으로 배포 가능한 파이프라인 완성

**무엇을:** 테스트 레이블 없이 정상 이미지의 Anomaly Score 분포만으로 임계값을 결정하는 퍼센타일 방식 적용 (90/92/95 퍼센타일 비교)

**왜 이렇게 했는가:** v4에서 FN이 많았던 근본 원인은 모델 성능이 아니라 임계값 결정 방식에 있었다. v4는 테스트 데이터를 본 뒤 사후적으로 최적 임계값을 계산했기 때문에 실제 현장에서는 사용할 수 없었다. v5에서는 정상 이미지 분포의 90~95 퍼센타일을 임계값으로 고정하여, 불량 이미지를 전혀 보지 않고도 배포 가능한 파이프라인을 완성했다.

**결과:**
- **FN: 36 → 4** (현장 적용 가능한 수준으로 감소)
- **AUROC: 0.994**
- FPS: 8.1 (이미지 1장당 약 200ms → 실시간 처리 가능)
- 임계값을 현장 요구사항에 따라 퍼센타일 수치로 손쉽게 조정 가능

## 주요 트러블슈팅

### 문제 1: v4 임계값 방식이 실제 배포에 사용 불가

**문제:** v4에서 각 조건의 최적 임계값을 계산하려면 테스트 불량 이미지의 레이블이 필요하다. 현장에서는 불량이 언제 어떻게 올지 모르는 상황이므로, 이 방식은 실험용 수치일 뿐 실제 적용 불가능하다.

**원인 분석:** "좋은 임계값"을 찾는 과정에서 의도치 않게 테스트 데이터의 정보를 사용하는 데이터 리케지(Data Leakage) 문제가 발생했다.

**해결 방법:** v5에서 정상 이미지의 Anomaly Score 분포만을 사용하는 퍼센타일 방식으로 전환. 불량 이미지 없이 임계값 설정 가능 → 배포 가능한 파이프라인 완성.

**결과:** FN 36 → 4, 현장 투입 가능한 수준 달성

---

### 문제 2: SoftPatch·Homomorphic 필터가 우리 데이터에서 효과 없음

**문제:** v3 실험에서 논문에서 성능 향상을 보인 SoftPatch 전처리와 Homomorphic 필터를 적용했으나 유의미한 개선 없음

**원인 분석:** SoftPatch는 배경 노이즈가 많은 이미지에 효과적이다. 버스바 이미지는 이미 ROI 크롭을 통해 배경이 제거된 상태였기 때문에 추가 전처리의 효과가 없었다. Homomorphic 필터도 조명 균일도가 비교적 높은 공장 환경에서는 개선 폭이 미미했다.

**해결 방법:** 두 기법을 v4 이후 실험에서 제외하여 실험 조합 수를 줄이고 탐색 효율 향상

**결과:** v4에서 더 의미 있는 하이퍼파라미터(해상도, Coreset 비율, kNN 수)에 집중 가능

---

### 문제 3: 30개 조건 중 절반에서 불량 데이터 0장

**문제:** 평가를 위해 불량 이미지가 필요한데, 조건의 절반 이상에서 불량이 0장이었다. 이 조건들에서는 성능 측정 자체가 불가능했다.

**원인 분석:** 실제 제조 현장에서 불량은 드물게 발생하며, 카메라-검사 조건에 따라 불량 출현 빈도가 다르다. 이것이 OCC 방식을 선택한 이유이기도 하지만, 동시에 평가의 한계가 된다.

**해결 방법:** 불량이 있는 조건에서의 성능을 기준으로 평가하고, 불량이 없는 조건은 "미검증 조건"으로 분류하여 결과 해석에 반영

## 배운 점

**기술적 인사이트**
- 임계값 하나가 "실험용 모델"과 "배포 가능한 모델"을 나누는 기준이 된다. v4와 v5의 AUROC 차이는 모델 성능이 아닌 임계값 결정 방식의 차이였다.
- 논문에서 효과적인 기법(SoftPatch, Homomorphic)도 데이터 특성이 다르면 효과가 없다. 논문을 맹신하지 않고 직접 실험으로 확인하는 과정이 필수다.
- OCC는 데이터 희소 문제의 현실적 해법이지만, 평가 자체도 불량 데이터가 필요하다는 구조적 한계가 있다.

**실무적 인사이트**
- 실험 설계는 "한 번에 한 변수만 바꾼다"는 원칙을 지켜야 원인과 결과를 연결할 수 있다.
- "성능이 좋다"는 것과 "현장에서 쓸 수 있다"는 것은 다른 문제다. 배포 가능성을 처음부터 설계 조건에 포함시켜야 한다.
- 3,000건 가까운 실험을 자동화하여 실행하면서, 실험 관리와 결과 추적의 중요성을 직접 경험했다.

## 향후 개선 방향

- **YOLO 기반 ROI 자동 크롭:** 현재 수동 크롭을 자동화하여 조건별 전처리 인력 절감
- **ReconPatch 적용:** 재구성 기반 이상탐지 모델로 전환하여 색상·형태 결함 탐지 정확도 향상
- **앙상블:** PatchCore + 재구성 모델 앙상블로 FN 추가 감소 목표
- **실시간 서버 배포:** 현재 구축된 추론 서버(`Patchcore_server/`)를 라인에 실제 연결
